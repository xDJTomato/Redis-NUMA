Redis CXL 基准测试 - Phase 1 内存占用调查
Executive Summary
================================================================================

调查问题:
  为什么 Phase 1 (Fill) 阶段 Redis 实际内存占用只有 ~4.5GB，而非预期的 ~9GB?

快速答案:
  ✓ 这不是 Bug
  ✓ 内存配置（maxmemory=3500mb）与测试规模（1M × 10KB）不匹配
  ✓ 导致 Redis 持续触发 LRU 淘汰，虽然写入了 200 万次操作，但只保存了 133 万条记录
  ✓ 观测到的"4.5GB"是 RSS，实际 used_memory = 3.5GB（maxmemory 限制）

================================================================================
关键发现
================================================================================

测试结果对比:

Test Case                    | 环境     | 写入操作    | 存储记录    | 驱逐记录   | 内存占用
-----------------------------|---------|------------|-----------|-----------|----------
bw_bench_20260417_001923    | CXL VM  | 2,000,434  | 1,331,993 | 668,441   | 3500MB
bw_bench_20260416_232722    | 宿主机   | 1,988,210  | 1,325,829 | 662,381   | 3500MB

结论: 两个环境表现一致 → NUMA 实现没有问题

================================================================================
根本原因分析
================================================================================

原因 1: maxmemory 硬限制
  • 脚本 run_bw_benchmark.sh 第 43 行: MAX_MEMORY="3500mb"
  • Redis 启动参数: --maxmemory 3500mb --maxmemory-policy allkeys-lru
  • 内存占用被限制在 3.5GB，永远不会超过

原因 2: 理论与配置的矛盾
  • 测试目标: 1,000,000 条 × 10,240 bytes = 10.3GB
  • 允许上限: 3.5GB
  • 缺口: 6.8GB

原因 3: Redis eviction 自动触发
  • 当 used_memory 接近 maxmemory 时自动触发 LRU 淘汰
  • YCSB 不断写入新记录
  • Redis 不断删除旧记录腾出空间
  • 最终: 写入 200 万次，保存 133 万条，驱逐 67 万条

用户观察到的"4.5GB":
  • used_memory = 3.5GB (Redis 内核计数，受 maxmemory 限制)
  • RSS = 4.2GB (系统报告的物理内存，包含碎片和系统开销)
  • 差异 = 0.7GB (内存碎片 + Redis 自身结构 + 操作系统开销)

================================================================================
解决方案
================================================================================

推荐方案 A: 增加 maxmemory (最直接)
  命令: ./run_bw_benchmark.sh --maxmem 11gb
  效果: 允许 Redis 使用充足内存，完整填充预期的 1M × 10KB 数据
  验证: evicted_keys 应为 0，used_memory ≈ 10.5GB

推荐方案 B: 调整数据规模匹配内存 (最对齐)
  修改: PHASE1_RECORDS=350000 (或 PHASE1_FIELD_LENGTH=3500)
  效果: 与 3.5GB maxmemory 完美对齐，无任何浪费
  优点: 配置清晰，便于调试和重现

推荐方案 C: 关闭内存限制 (激进)
  修改: 注释掉 --maxmemory 和 --maxmemory-policy 参数
  效果: Redis 不受限制，充分测试 NUMA 内存池性能

技术细节
================================================================================

Redis String 对象内存布局:
  redisObject (Key)      : 16 bytes
  SDS Header (Key)       : 30 bytes + "user000000000"
  redisObject (Value)    : 16 bytes
  SDS Header (Value)     : 17 bytes + 10,240 bytes
  字典项开销             : ~16 bytes
  ────────────────────────────────
  总计                   : ~10,330 bytes ≈ 10.1 KB/条

maxmemory 与能存储的记录数:
  3,500 MB ÷ 10.1 KB = 346,534 条 (理论最大)
  实际存储: 1,331,993 条 (由于 eviction 策略的动态特性更多)

eviction 过程:
  • Redis 在内存写入时不是"一次性检查"而是"连续监控"
  • 当 used_memory > maxmemory * 0.75 时开始扫描
  • 删除最少使用的键
  • 循环执行直到 used_memory < maxmemory
